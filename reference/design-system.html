# A field guide for cleaner internal tools

A short guide for shipping technical interfaces that feel fast and stay legible at 2am. The system biases toward keyboard-first interaction, dark-by-default surfaces, and prose that reads like a thoughtful colleague — not a vendor.

Most teams over-design tools and under-design content. This guide is about getting the second part right: typography, hierarchy, and the connective tissue between your dashboards, consoles, and docs.

Note

Markdown in this system is opinionated. Tables don't have row stripes, bold doesn't get heavier than 600, links don't go blue. Lean into it.

# Getting started

You need three things to begin: a markdown file, a parser of your choice, and patience for round two of your own writing. The system handles everything after parse.

# Installation

Wrap parsed HTML output in .markdown. That's the whole API. Your existing tokens and components cascade in automatically. 

import { marked } from 'marked';

const html = marked.parse(myMarkdown);
document.querySelector('article').innerHTML = html;
document.querySelector('article').classList.add('markdown');

# Customization

A few decisions are worth making explicitly:

  1. Whether tables stay full-width or constrain to prose width.
  2. How aggressive the heading scale gets — defaults work to h4, beyond that depends on your content.
  3. Whether to enable strikethrough globally or scope it to changelog files.

# Features

Callouts

Five variants matched to the status palette:

Tip

Use callouts sparingly. Three in a row and they become wallpaper — every one less impactful than the last.

Important

Anchor links require ids on every heading. If you're parsing markdown server-side, slug the heading text and set id on the element.

Warning

Long code samples will overflow horizontally on narrow viewports. The scroll behavior is by design — wrapping breaks alignment in shell snippets.

Caution

Never paste untrusted markdown into innerHTML without sanitization. Use DOMPurify or your parser's safe mode.

# Browser support

The system targets modern evergreen browsers1. CSS uses :where(), color-mix(), and CSS nesting in a handful of newer components2. Fallbacks are not provided.
